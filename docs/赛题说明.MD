任务介绍
在真实科研与模型研发过程中，实验优化往往不是一次性完成的，而是围绕“提出配置—运行试验—读取反馈—决定下一步”的闭环持续展开。研究者通常无法提前获得完整监督，而只能看到有限的中间反馈，例如验证集指标、训练耗时、资源占用、失败日志等。如何在有限预算内利用这些稀疏反馈持续修正实验策略，是自动化科研与智能体研究中的核心问题之一。

本赛题面向Agent 长程推理与自动化实验控制，聚焦有限预算、稀疏反馈、禁止并行约束下的自动化实验智能体构建。参赛者需要设计一个能够在多轮交互中持续优化实验决策的 agent：读取历史试验结果，归纳有效经验，决定下一轮模型、超参数或训练策略，并在预算耗尽前输出尽可能优的预测结果。

为保证评测标准化与可复现性，赛题采用金融场景下的图学习任务作为测试任务，并设置两个具体子任务：

（1）产品分类任务：以图节点分类的形式进行评测；

（2）产品推荐任务：在用户-产品的二部图上的图链接预测问题。

两个子任务都面向稀疏监督场景，即测试节点/用户可利用的邻域或历史信息较少。比赛重点不在于手工调参或单一模型结构本身，而在于 Agent 是否具备以下能力：在多轮实验中维护有效的实验记忆；从稀疏反馈中提炼规律并更新后续策略；在预算约束下进行合理的探索、利用与停止决策；从不同数据集的实验过程中总结可迁移经验用于新的图任务。

为便于理解，可以将本赛题看作一个受控的金融模型研发沙盒。以新用户金融产品推荐为例，Agent 首先需要根据已有用户行为、产品特征和历史交互结果生成一个初始实验方案；随后根据有限反馈判断当前方案的不足，例如部分新用户预测不稳定、某类产品召回不足、模型训练成本过高，或验证结果与线上反馈不一致；接着，Agent 需要自主决定下一轮实验方向，例如调整特征使用方式、改变模型结构、修改训练策略、重新分配实验预算，或在收益有限时提前停止。经过多轮迭代后，Agent 需要在预算耗尽前选择当前最优方案并生成最终提交结果。

因此，本题是一个“受控机器学习实验环境中的长程决策问题”，也是一个兼具研究性与工程性的综合挑战题。赛题不仅关注最终预测结果的准确性，也关注 Agent 系统在有限预算下如何组织多轮实验、利用反馈信息、形成优化策略并逐步改进预测器。

二、任务目标
本赛题围绕“稀疏反馈下的自动化实验控制”展开，参赛者需要面向给定的图学习任务，设计一个能够自主开展多轮实验的智能体系统。该系统需要在资源受限条件下，自动完成“读取历史实验结果—分析反馈信号—决定下一轮实验配置—持续迭代优化—输出最终预测结果”的完整流程。

本赛题包含两个具体子任务：

产品分类任务：参赛者需要根据给定图数据（含产品节点特征与节点间连接关系），对测试节点进行类别预测；

产品推荐任务：参赛者需要根据给定用户产品交互序列、用户特征和产品特征，为每个测试用户预测一个按置信度排序的产品列表。

两个子任务都面向典型的稀疏监督场景，在产品分类任务中，参赛者需要对连接关系较为稀疏的节点进行分类；在产品推荐任务中，参赛者需要对仅具有少量交互历史的新用户进行推荐。因而参赛方法不能依赖密集的监督信号，而需要在有限反馈、有限预算和多轮实验过程中，逐步归纳有效经验并优化决策策略。

参赛者需要构建一个自动化实验 Agent系统。对于每个待求解的任务与数据集，Agent 需要自主决定尝试的预测器模型结构、超参数配置、训练策略或停止时机，并在有限预算内不断改进结果，最终得到预测器来完成预测任务。在比赛提交阶段，参赛者需要按照赛题要求提交预测结果文件，平台将根据预测结果进行自动评测。进入复审阶段的队伍还需提交完整代码，以证明其方法能够在规定环境中稳定复现。

本赛题鼓励参赛者结合图学习、自动化搜索、实验记忆、规划与反思、多 Agent 编排、RAG，以及主办方允许的 Qwen 系列模型 API 等技术来解决问题。考虑到复现阶段对代码体积和运行环境的要求，参赛者应优先设计轻量、稳定、可审计的预测器模型与 Agent 系统，而不是依赖难以复现的本地大模型。

为帮助参赛者理解任务形式，赛题将提供公开评测数据、统一输入输出格式、提交模板及相关说明文档。参赛者需要基于这些材料，构建一个能够在未知图任务上持续优化实验决策并输出高质量预测结果的自动化实验系统。

三、任务数据
本赛题采用 A/B 榜机制，A 榜将提供初赛测试集，用于模型调试、提交格式校验与阶段筛选；B 榜将提供复赛测试集，用于代码提交阶段的自动评测排名。

A 榜、B 榜都各包含 1 个产品分类数据集以及1 个产品推荐数据集。A/B 榜在任务形式、输入输出接口、运行约束和评测指标上保持一致，区别主要在于数据划分、测试目标以及是否用于最终排名。

（三） 校验规则及释义
产品分类任务校验规则如下：

节点编号规则：所有节点编号均为从 0 开始的整数编号。

标签规则：labels 中，训练节点位置为公开标签；测试节点位置统一为 -1，不代表真实类别。

训练集规则：train_idx 为可用于训练/验证的节点集合，选手需自行制定内部验证策略。

测试集规则：test_idx 为待预测节点集合。选手需对这些节点输出类别编号，并严格按照提交模板中的顺序填写。

类别编号规则：提交的 label 必须为合法整数类别编号。A 榜类别范围为 0 到 9。B 榜类别范围以 B 榜数据说明文件为准。

产品推荐任务校验规则如下：

用户编号规则：提交文件中的 uid 必须与官方 test.csv保持一致，不得删除、增加或重排用户行。

产品编号规则：prediction 中的 item id 必须来自对应数据包 item.csv 的 iid 候选集合。

排序规则：prediction 中的 item id 需按置信度从高到低排序，总计10个，使用英文逗号分隔。

去重规则：同一用户的推荐列表中不应出现重复 item id；若出现重复、非法 id 或缺失预测，平台将按评测脚本规则处理，可能导致该用户得分降低。

（二） 评分规则和脚本
代码提交阶段自动评测
平台将在统一环境下对选手提交结果进行自动评测。产品分类任务和产品推荐任务分别计算得分，最终榜单分数由二者加权得出。

产品分类任务采用 Accuracy 作为评测指标，得分为测试节点上的平均预测准确率。

产品推荐任务采用 NDCG@K 作为评测指标，默认 K=10。即对每个测试用户，检查隐藏目标 item是否出现在预测列表中以及排位情况；再对全部测试用户取平均。最终榜单分数为节点分类任务分数Score_cls和推荐任务分数Score_ndcg的平均：

Score_final = 0.5 * Score_cls + 0.5 * Score_ndcg

A榜与B榜的上述分数实行独立核算。其中，B榜自动评测成绩将作为评定榜单排名及确定报告提交名单的依据。

决赛路演答辩将重点考察参赛方案的创新性、完整性、可复现性与自动化实验控制能力。进入答辩阶段的队伍需要结合过程日志与实验记录，清晰展示 Agent 如何在多轮实验中读取反馈、分析失败原因、调整模型或超参数、形成下一轮优化策略，通过具体示例说明其方法的有效性、稳定性和可靠性。评审将综合考虑最终预测效果与实验过程质量评价参赛系统。

（三）代码审核
B 榜成绩排名前 15 名的参赛者需提交完整的代码和复现文档，包括但不限于以下内容：Agent 系统主流程代码、预测器训练与推理代码、配置文件、运行说明文档、过程日志文件、必要的实验记录与复现脚本。

代码需使用 Python 语言编写，并附上适当的注释，以便于评审团进行审核。代码需能够完整复现参赛者在产品分类任务和产品推荐任务上的预测结果，并通过脚本命令实现稳定复现。复审阶段将检查代码运行过程、过程日志与最终预测结果之间的一致性。若过程日志无法与实际代码逻辑、实验记录或提交结果相互印证，主办方可要求参赛者进一步说明；存在伪造日志、补写轨迹或人工干预实验过程等情况的，成绩将被判定无效。

所有使用到的模型、数据与 API 均需符合比赛规则；若调用主办方允许的模型服务，应在复现文档中说明用途与调用位置。

模型体量与可复现性要求：提交包总大小不得超过 1GB，选手应优先采用轻量、可审计、可稳定复现的预测器。若使用本地大模型，需自行承担无法通过体积限制或复现审核的风险。

代码中不得包含任何绕过自动化实验过程、直接注入任务答案、或以不可复现方式生成结果的逻辑，否则成绩将被判定无效。 包括但不限于以下情形：

固定结果填充或伪造预测：直接使用预先写死的默认标签、固定排序列表、固定分布采样结果或其他与输入无关的占位结果进行填充。

将完整预测器或核心解法直接硬编码进 Agent 提示词：不允许将完整的预测器实现、面向特定任务的整套最优代码、关键超参数组合、固定模型结构或固定训练流程，以长段代码、伪代码或等价规则的形式直接写入Agent 提示词中，使Agent 实际上不再进行自主设计、分析与迭代优化，而只是“复述”预埋方案。

针对特定数据文件或样本的硬编码规则：不允许根据数据集文件名、样本顺序、节点编号、用户编号、item 编号或其他可识别标记，编写专门的分支逻辑、查表逻辑或映射规则，直接输出特定预测结果。

利用评测反馈反推测试答案：不允许通过反复试探提交、构造特殊提交样本、分析排行榜反馈、比较多次评测差异等方式，对测试集标签或目标结果进行反向推断。

利用非规定信息源构造答案：不允许通过未备案外部服务、隐藏数据、私有标注、人工整理的测试集答案库，或其他不被允许的信息源直接生成或辅助生成测试结果。
