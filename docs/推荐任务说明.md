# 推荐任务说明

> 面向推荐任务负责人（REC_TASK）的任务理解文档。基于实际 A 榜数据撰写。

## 一句话概括

给定一个用户最近点过哪些产品（行为序列），预测他**接下来最可能点哪个产品**，并给出一个按可能性排序的 top-10 产品列表。

本质上这是一个**序列推荐 / 下一个 item 预测**任务，评测用 **NDCG@10**。

---

## 数据里到底有什么

比赛给了 5 个文件，用真实样例逐个拆解：

### 1. `train.csv` —— 训练集（4万用户，**带答案**）

```
uid,      target_iid, item_seq_raw,                                   item_seq_dedup,        item_seq_counts
u000001,  i001455,    "i000968,i000968,...,i001455",                   "i000968,i001455,...", "i000968:18,i001455:14,..."
```

对每个用户有 4 个字段，**关键是 `target_iid`（答案）**：

| 字段 | 含义 | 例子 |
|---|---|---|
| `uid` | 用户 | u000001 |
| `target_iid` | **这个用户接下来真实点击的产品（要预测的目标）** | i001455 |
| `item_seq_raw` | 用户最近的**原始点击序列**（含重复，按时间顺序） | i000968,i000968,i001455,... |
| `item_seq_dedup` | **去重保序**后的序列 | i000968,i001455,... |
| `item_seq_counts` | 每个 item 被点过几次 | i000968:18,i001455:14,... |

注意看 u000001：他的 `target_iid = i001455`，而 i001455 在他的历史里已经出现过14次。**这就是这个数据的核心规律：用户倾向于反复点自己常点的产品。** 这一点直接决定了方法设计。

### 2. `test.csv` —— 测试集（1万用户，**隐藏答案**）

```
uid,      item_seq_raw,    item_seq_dedup,    item_seq_counts
u000009,  (空),            (空),              (空)
u000014,  "i000832,...",   "i000832,i000476", "i000832:8,i000476:2"
```

和 train 完全一样，**唯独没有 `target_iid` 列**。任务就是为这1万个用户把那个被隐藏的 target 预测出来。

特别留意 **u000009：历史全空**。这就是赛题强调的"稀疏反馈"——有相当一部分测试用户是**冷启动用户**（什么历史都没有），只能靠用户特征去猜。这是任务的一大难点。

### 3. `user.csv` —— 用户侧匿名特征（8个分类特征）

```
uid,      u_cat_01,...,u_cat_08
u000001,  6,2,4,31,5,3,2,2
```

每个用户8个匿名分类特征。对**冷启动用户**（历史为空）来说，这是唯一能用的信息——靠"这类用户通常点什么产品"来推荐。

### 4. `item.csv` —— 产品侧匿名特征 + **候选集**

```
iid,      i_cat_01,i_cat_02,i_cat_03,i_bucket_01
i000001,  0,0,0,3
```

**重点：`item.csv` 里全部的 iid 就是候选产品池**（A榜2156个 / B榜14065个）。每个推荐都必须从这个池子里出。

### 5. `sample_submission.csv` —— 提交格式

```
uid,      prediction
u000009,  "i000001,i000002,i000003,...,i000010"
```

每个测试用户给 **10个 iid，逗号分隔，按置信度从高到低排**。

## 任务流程图

```
        用户历史序列 (test.csv)                要输出的
        ┌──────────────────┐                  ┌─────────────────────┐
   ───▶ │ item_seq_raw /   │  ── 模型 ──▶     │ prediction: top-10  │ ───▶ 提交
        │ dedup / counts   │                  │ 按置信度排序的iid   │
        └──────────────────┘                  └─────────────────────┘
              (+ user.csv 特征, + item.csv 特征)
              历史为空时(冷启动) 只能用 user.csv 特征
```

训练时用 `train.csv`，那里有答案 `target_iid`，让模型学会"历史序列 → 下一个item"的映射。

---

## 怎么评分：NDCG@10

对每个测试用户：
- 真实答案 `target_iid`（被隐藏的）只有 **1 个**
- 给了 10 个候选，排了序
- 如果真实答案出现在列表里，且排第 `i` 位（从0开始），得分 = `1/log2(i+2)`
  - 排第1位 → 1.0（满分）
  - 排第2位 → 0.63
  - 排第10位 → 0.289
  - 没出现在 top-10 → 0 分
- 所有1万用户取平均

**含义**：不仅要"猜中"，还要"排在前面"。因为答案只有1个、列表10个，所以这是个**排名任务**——不需要给所有2156个产品都精准打分，只要让真答案进前10、且尽量靠前就行。

---

## 这个任务的三大难点（决定方法设计）

从现有 `SOTA_predictor` 的诊断里看到的真实情况：

1. **历史极短**：测试用户历史中位数只有3个item，很多模型"喂不饱"。
2. **冷启动占比高**：~35% 测试用户历史为空，纯靠用户特征猜，天花板很低。
3. **重复即答案**：约 **56% 的真实答案已经在用户自己的历史序列里**。所以"用户最近反复点的产品"是最强的单一信号——很多答案就是"再点一次他常点的"。

---

## 现有方案做到什么程度

`SOTA_predictor/rec` 里的 `ranker_hybrid`：
- 把 pop（流行度）、repeat（重复）、EASE（共现）、Markov（转移）、htarget（历史→目标关联）、cold-ridge（冷启动）这些信号**全部算成特征**，喂给一个 LightGBM 排序器学权重
- 冷启动用户单独用"用户特征→目标产品"的岭回归
- 用诚实协议（按测试集的短历史分布截断）验证，**NDCG@10 ≈ 0.50**

## 规模与约束（A榜 → B榜）

| 指标 | A榜 | B榜 |
|---|---|---|
| 用户数 | 50000 | 50000 |
| 训练用户 | 40000 | 40000 |
| 测试用户 | 10000 | 10000 |
| 产品(item)数 | 2156 | **14065** |

B榜产品池扩大 ~6.5 倍，方法需可扩展（避免对全 item 做重型计算）。任务形式、字段、提交格式、评测指标 A/B 榜一致。

## 运行约束

- 单数据集总运行时长（Agent 生成预测器 + 预测器推理）≤ 2 小时
- 参考硬件：单张 NVIDIA 4090 24GB 或同等
- **禁止并行运行多个实验进程**
- 预测器须轻量、可复现（提交包 ≤ 1GB，不建议本地大模型）
- 允许 Qwen3.5/3.6 系列模型 API + Qwen text-embedding-v4
