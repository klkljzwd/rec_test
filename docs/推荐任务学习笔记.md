# 推荐任务学习笔记（关键理解点）

> 记录推荐任务负责人在理解任务与特征构建过程中，与AI对话沉淀的关键认知。
> 配合 `推荐任务说明.md`（任务总览）与 `build_features.py`（特征构建代码）阅读。

---

## 一、任务本质

- **序列推荐 / 下一个 item 预测**：给定用户历史点击序列，预测他接下来最可能点的产品，输出按置信度排序的 top-10。
- **评测 NDCG@10**：每用户真实答案只有 1 个，排第 i 位得分 = 1/log2(i+2)（第1位1.0、第2位0.63、第10位0.289，未进top-10得0）。所有1万测试用户取平均。
- **本质是排序任务**：不需要给所有item精准打分，只要让真答案进前10且尽量靠前。

## 二、数据核心规律（决定方法设计）

来自实际统计 + 现有 `SOTA_predictor` 诊断：

1. **历史极短**：测试用户历史中位数仅 3 个item；L=0(冷启动)占35.1%，L=3占44.4%，90%用户历史只有0~3个item。
2. **冷启动占比高**：35.1% 测试用户历史全空，只能靠 user.csv 特征猜，天花板很低。
3. **重复即答案**：约 **56% 的真实 target 已在用户自己的历史序列里** → repeat/recency 是最强单一信号。
4. **候选集实际很小**：A榜2156个item里只有 **235个** 曾作为训练 target 出现过 → 预测时真正有竞争力的候选就这235个。
5. **现有方案**：`ranker_hybrid`（LGBM LambdaRank + 冷启动ridge），诚实协议 NDCG@10 ≈ 0.50。

## 三、特征构建：把问题转成排序学习

### 核心机制
- 原始数据是"一用户一条记录"，特征表展开为 **一行 = 一个 (用户, 候选item) 对 + 标签**。
- **正样本**（1个）：用户真实 target，label=1。
- **负样本**（N个）：非 target 的 item，label=0。
- 排序的**作用域是单个用户内部**：LGBM LambdaRank 用 `group` 参数按 uid 分组，模型只在同一用户的 N+1 个候选之间学排序（跨用户比毫无意义）。
- 模型输出每个候选的**标量得分**，组内按分排序；训练目标让 label=1 的得分尽量排最前。

### 训练 vs 预测对接
```
训练: 每用户 1正+N负 → 学打分函数 f(用户特征, item特征)=分
预测: 测试用户面对真实候选池(那235个)，对每个候选算f，按分排序取top-10
```
训练时每用户只采 N+1 候选是为省算力+构造对照；预测时对候选池所有有竞争力item打分。打分函数须能泛化到训练时未见过的候选。

### 负样本
- **是什么**：不是该用户 target 的 item，作用是教会模型"什么不该排前面"。
- **为什么必须有**：光给正样本模型只会"认target长什么样"，学不会"在多个候选里分高下"。负样本提供排序信号（衬托出"分高的该排前面"的规律）。
- **为什么从235个里采、不从2156里采**：从全量采会混入1921个永不当target的"废item"，模型在简单题刷分到难题无用；从235个真实竞争者采，负样本是"本可能是答案但这次不是"的有力对手，逼模型学真正的候选间排序。
- **数量**：正负比例失衡模型会偷懒，一般每正样本配 5~20 负（默认 neg=20）。

### 特征分6组（23特征列）
| 组 | 特征 | 来源 | 受截断影响 |
|---|---|---|---|
| A 用户-候选历史交互 | in_hist/count/count_norm | train.csv序列 | ✅ |
| B item全局 | popularity/is_known_target | 全局统计 | ❌ |

## 四、为什么不能直接用原始序列当特征

1. **结构上**：特征表行是(用户,单个item)，每行须是定长数值向量；原始序列是变长列表(0~174)，排序器读不懂变长token。
2. **信息上**：原始序列有信号但"埋"在里面，不感知item语义的排序器无法自动归纳（不会自己总结"点了3次、是最近点的、和X共现"）。聚合特征 = 提前把规律提炼成模型能直接读的数值。
3. **但原始序列不是没用**：序列模型(GRU4Rec/SASRec)直接吃原始序列自己学表征。本任务历史极短+冷启动多，序列模型难从3个item学稳定规律，手工特征(repeat/htarget)更稳更吃到"56%重复"红利。
4. **现实约束**：赛题要求轻量(≤1GB)、2小时内、禁并行、可审计；聚合特征是闭式解/统计量，CPU秒级确定性强，序列模型重且难复现。两类方法互补 → 方法候选池同时考虑序列模型作补充。

## 五、截断（Truncation）

### 截断的是什么
**截断的就是用户过去点击的物品列表（历史序列）**。取最近 L 个 run-length item。
```python
raw = [i000968,i000968,i001455,i001191,i000968,i001455,...]  # 完整历史
截断到 L=3 → [i001191, i000968, i001455]  # 最近3个
```
截断后 `ded`(去重序列) 和 `cnt`(次数) 都变了。

### 影响哪些特征
- **受影响（依赖该用户历史的特征）**：in_hist / count / count_norm / repeat / ease / markov / htarget。同一(用户,item)对，截断与否特征值完全不同（如 count 可能从14变2）。
- **不受影响（全局/固有特征）**：popularity / is_known_target / u_cat / i_cat / i_bucket / ufeat_target_cond。这些本就不依赖历史长度。

### 为什么必须截断
**训练时模型看到的历史长度必须和测试时一致，否则学出的规律在测试时失效（分布偏移 distribution shift）。**
- 不截断：训练用完整长历史（count能到50+），模型学到"count=14→正样本"；测试只有3个item历史，count最大2~3，规律取不到 → 训练分虚高、测试分掉。
- 截断到短历史：训练集 count 取值范围(0~3)与测试集对齐，学到的"count大→更可能target"规律在测试时才用得上。

### 没有测试集能否截断
**能。截断本质是"按短历史训练匹配稀疏预测场景"，由任务性质(稀疏反馈)决定，不由有无测试集决定。**
- 有测试集长度分布：截断采样分布与测试完全对齐，验证分可信、效果最优。
- 无测试集分布：用①覆盖式截断(取L=0,1,2,3,5,10…各训一遍，数据增强)或②从训练集自身长度分布采样。代价是验证分与真实测试分有偏差，但方向正确，仍避免"只学长历史"的致命错误。
- **关键细节**：截断必须用 run-length 重建(`rld`)，不能直接切 item_seq_dedup——测试集的 dedup/counts 是按"连续重复合并"规则从raw生成的，直接切dedup会得到错误counts。
- 即便测试集历史很长，截断仍有价值（正则化，让模型学"用少量信息也能预测"的鲁棒性）。

## 六、规模与运行约束（A→B榜）
| 指标 | A榜 | B榜 |

- 单数据集总时长 ≤ 2小时；参考4090 24GB；**禁并行**；提交包 ≤1GB；允许 Qwen3.5/3.6 API + Qwen text-embedding-v4。

## 速记口诀
- 任务 = 短历史/冷启动下预测下一个item，输出top-10排序。
- 特征 = 每用户展开成 N+1 个(用户,item)排序样本，正1负N。
- 负样本 = 从235个真实竞争者里采，提供排序信号。
- 截断 = 砍短历史使训练/测试长度分布对齐，只影响依赖历史的特征。
- 现有 ranker_hybrid ≈ 0.50，盲区在端到端协同/序列表征(可补)。
